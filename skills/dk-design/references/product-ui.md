# Product UI kernel

Read for dashboards, admin tools, forms, repeated workflows, tables, or state-heavy application screens.

- Put the primary task and next decision before decoration. Treat navigation, status, and destructive actions as separate hierarchy roles.
- Preserve real data density. Do not turn a useful product into a sparse landing page or wrap every value in an interchangeable card.
- Design loading, empty, partial, error, success, disabled, focus, permission, overflow, long-copy, and reduced-motion behavior where relevant.
- Keep forms close to the decision they support. Use persistent labels, specific help, field-level errors, a submission error summary when needed, and visible success confirmation.
- On narrow screens, preserve task order rather than shrinking the desktop composition. Move secondary analysis after the action, keep targets usable, and avoid horizontal overflow unless the data model truly requires it.
- Tables need scan anchors, explicit units, meaningful alignment, sorting/filtering states, empty results, and a deliberate small-screen strategy.
- Dashboards need a decision hierarchy. Metrics without an owner, time range, comparison, or action are decoration and must not be invented.
- Reuse accessible product components, but do not let a component library choose the art direction. Tokens carry identity; components carry behavior.
